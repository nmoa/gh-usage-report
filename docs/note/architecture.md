# アーキテクチャとディレクトリ構成

本ドキュメントでは、本プロジェクトのディレクトリ構成および各パッケージの責務について説明します。

## ディレクトリ構成

リファクタリング後のディレクトリ構成は以下の通りです。

```
gh-usage-report/
  main.go                  # エントリーポイント
  AGENTS.md                # 開発エージェント向けのルール定義
  README.md                # プロジェクトの概要と使用方法
  feature.md               # 満たすべきフィーチャー一覧
  docs/                    # ドキュメント類
    note/
      architecture.md      # 本ドキュメント
  internal/
    cli/                   # コマンド定義と実行フロー（副作用の接続点）
      root.go              # ルートコマンド定義
      aggregate.go         # aggregate サブコマンド定義
    aggregate/             # CSV パースおよび集計ロジック（純粋関数）
      aggregate.go         # 集計処理の実装
    billing/               # 請求期間の計算ロジック（純粋関数）
      cycle.go             # 請求サイクルから対象日付を算出する処理
    api/                   # Usage Reports API クライアント
      octokit.go           # API 呼び出しの抽象化
      types.go             # API リクエスト・レスポンスの型定義
    download/              # CSV ダウンロードとリトライ処理
      download.go          # HTTP ダウンロードとリトライ処理の実装
```

## 各パッケージの責務

### main（ルート）
アプリケーションのエントリーポイントです。`internal/cli` パッケージの `NewRootCmd` を呼び出してコマンドを実行するのみの最小限の構成となっています。

### internal/cli
CLI コマンド（ルートコマンドおよび `aggregate` サブコマンド）の定義と実行フローの制御を担当します。
外部 API クライアント、ファイルシステム、および標準出力などの副作用を伴う依存関係をまとめ、各ドメインロジックへと接続する責務を持ちます。

### internal/aggregate
ダウンロードされた Usage CSV ファイルのパース、および指定された単位（組織、コストセンター、ユーザー）での集計とソート、再出力用 CSV フォーマットへの変換を行います。
このパッケージ内のロジックは副作用を持たない純粋関数として構成されており、すべてのロジックに対して単体テストが網羅されています。

### internal/billing
CLI オプション（年、月、請求開始日）から、Usage Reports API に指定する正確な請求期間（開始日・終了日）を計算するドメインロジックを提供します。
閏年や存在しない日付のハンドリングなどを含む純粋な日付計算処理であり、独立した単体テストによって品質が担保されています。

### internal/api
GitHub Usage Reports API と通信を行うためのクライアント（Octokit）と、関連するデータ構造を提供します。
GitHub CLI が提供する REST クライアントを利用して、レポートの生成要求および進捗の取得を行います。

### internal/download
レポート生成完了後に、提供された URL から実際の CSV データをダウンロードする処理を担当します。
ネットワークの一時的なエラーに対する指数バックオフを用いたリトライ処理などがカプセル化されています。
`http.Client` が未指定の場合は、パッケージ初期化時のグローバル状態を排除するために、利用時にデフォルトの HTTP クライアントを遅延生成する設計となっています。
